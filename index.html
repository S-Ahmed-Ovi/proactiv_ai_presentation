Executive Overview

The Problem We Solve

Building recommendation infrastructure from scratch takes 6–12 months and requires specialized ML teams. ProactiveAI changes that equation entirely.

Business Problem
Organizations lack plug-and-play recommendation infrastructure. Most SaaS products and enterprise platforms cannot surface personalized content without investing heavily in custom ML teams and months of build time.
Target Market
E-commerce platforms, streaming services, content publishers, government agencies, and any business requiring personalized user experiences at scale — across domains.
Business Value
Increases user engagement, reduces churn, and drives conversion by surfacing the right item to the right user at the right time. Per-tenant model isolation ensures data privacy with zero co-mingling.
Expected Outcomes
Measurable uplift in CTR, session depth, and revenue per user. Composite scoring (70% accuracy + 20% speed + 10% scalability) automatically guides optimal model selection per tenant.

Core Challenges Addressed
Cold-Start Problem — New users and items receive poor recommendations without historical data. Our hybrid models mitigate this with content-based fallbacks.
Model Drift — Performance degrades as preferences shift. Our automated retraining pipeline detects feedback changes and retrains on schedule.
Multi-Tenancy — Shared systems create data leakage risks. We enforce per-tenant isolation at every layer: auth, model storage, and serving.
Domain Mismatch — One-size-fits-all models fail across verticals. The 21-algorithm registry covers CF, content, graph, hybrid, and neural paradigms.
Discovery Friction — Users cannot find relevant items without transparent personalization. Explainability endpoints surface reasoning for every recommendation.
Manual Tuning — ML teams waste weeks on hyperparameter search. Optuna TPE sampler automates this with SQLite-persisted studies per tenant per algorithm.
System Overview

End-to-End Platform

Every component in the system is purpose-built for multi-tenant isolation, automated model lifecycle management, and sub-100ms inference.

React Frontend — SaaS Dashboard
React 18 + Vite + Tailwind CSS + Recharts + Framer Motion. Ships 14 feature pages: Dashboard, Training, Benchmarks, Registry, Deployment, API Playground, Analytics, and more. Tenant-scoped sessions via JWT stored in localStorage. Framer Motion delivers declarative animations matching enterprise SaaS quality.
FastAPI Backend — REST API Gateway
FastAPI 0.110+ with Uvicorn ASGI server. 12 route modules: /recommend, /training, /models, /benchmarks, /analytics, /batch, /scheduler, /explain, /similar, /context, /deployment, /auth. Dual-auth via Bearer tokens (HMAC-SHA256 signed) and API keys. Redis-backed rate limiting with per-tenant metering.
Recommendation Engine — Core Intelligence
RecommendationStrategyService routes requests to: top-model (best single algorithm), weighted-ensemble (blended scores), or explicit single-algorithm strategies. RecommenderEngine scores users against loaded models, excludes seen items, and returns a ranked list with item scores. Latency budget: Auth (~1ms) + Model Load (~5ms cached) + Scoring (~10–50ms) + Filtering (~2ms) = sub-100ms P99.
Algorithm Registry — 21 Models
ALGORITHM_REGISTRY maps 21 algorithms with rich metadata: feedback type (explicit/implicit/both), scalability tier, domain affinity, and production readiness score. Algorithms span: SVD, NMF, ALS, BPR, EASE, SLIM, P3Alpha, RP3Beta, WARP, KNN variants (user/item/cosine/tanimoto), LightFM, Autoencoder, Content TF-IDF, Factorization Machines, domain-specific e-commerce and movie models.
Training Pipeline — Automated MLOps
training_pipeline.py orchestrates: FeedbackDetector → DatasetAnalyzer → Algorithm Eligibility Filter → Optuna TPE Tuning → BenchmarkEngine Evaluation → Composite Scoring → Model Promotion. Runs as an async background job — never blocks serving. Results stored in tenants/{id}/last_training_result.json with full leaderboard and per-model metrics.
Embedding Service — Semantic Search
Sentence-Transformers (all-MiniLM-L6-v2, 384-dim) generates semantic embeddings for item metadata. Embeddings are indexed per-tenant in OpenSearch with kNN plugin (HNSW graph, cosine similarity, ef_search=100). Enables semantic similarity search beyond interaction-based collaborative filtering — especially powerful for cold-start items with rich metadata.
Smart DB CSV Builder — LLM Data Ingestion
LLM-powered feature that uses a planner model to generate SQL queries from natural language descriptions, then builds CSV datasets from connected databases (PostgreSQL, MySQL, SQLite, MongoDB). Enables non-technical users to connect enterprise data sources without writing SQL. Connector support: psycopg2, pymysql, sqlite3, pandas CSV, and a NoSQL stub for MongoDB/DynamoDB.
Auth Gateway — Security Layer
JWT via python-jose (HMAC-SHA256 signed, configurable TTL). API key resolution via TenantManager.resolve_api_key() — hash-based comparison, no raw keys stored. 11 scopes from recommend:read to admin:write enforced by @require_auth() decorator on every route. Redis-backed rate limiting with per-API-key counters and TTL.
Composite Scoring Formula
composite_score = 0.70 × normalize(primary_metric) // NDCG@K or Precision@K + 0.20 × normalize(training_time, minimize=True) + 0.10 × scalability_score // 0.94 for unlimited-scale algos // Example: Ecommerce-Popularity on Tenant 002 composite = 0.70 × 1.0000 + 0.20 × 0.9896 + 0.10 × 0.94 = 0.9919

AI / ML Engine

Recommendation Paradigms

Six distinct algorithmic families covering the entire spectrum of modern recommendation approaches — from classic matrix factorization to neural semantic search.

Collaborative Filtering
SVDALSBPRNMF
Learns latent factors from user-item interaction matrices. SVD decomposes the rating matrix; ALS and BPR optimize for implicit feedback. Best for: dense interaction data with known users and items. Limitation: cold-start problem for new entities.
Content-Based Filtering
TF-IDFCosine Sim
Matches item attributes to user preference profiles using TF-IDF vectorization and cosine similarity. Best for: cold-start items with rich metadata, domain-specific matching. Limitation: over-specialization; misses serendipitous discoveries.
Graph-Based Models
P3AlphaRP3BetaEASE
Propagates signals through the user-item bipartite graph. P3Alpha and RP3Beta perform random walks with normalization. EASE derives a closed-form item-item similarity matrix. Best for: sparse interaction matrices; EASE excels at top-K precision. Limitation: EASE limited to ~8K items (O(n³) inversion).
Hybrid Models
LightFMFMAutoencoder
Combines CF signals with content features. LightFM fuses user/item embeddings with side-information features via BPR or WARP loss. Factorization Machines model feature interactions explicitly. Best for: cold-start mitigation + accuracy. Why selected: best balance between accuracy and cold-start robustness.
Neural / Deep Models
AutoencoderWARP
Learns non-linear user and item representations via deep encoders. WARP (Weighted Approximate-Rank Pairwise) optimizes directly for top-K ranking metrics. Best for: complex interaction patterns and high-dimensional feature spaces. Roadmap: DIN, SASRec, Transformers4Rec in Long Term phase.
Embedding-Based (Semantic)
Sentence-BERTOpenSearch kNN
Transforms item metadata into 384-dimensional dense vectors via all-MiniLM-L6-v2. Enables approximate nearest neighbor (ANN) search via OpenSearch HNSW graph — cosine similarity at scale. Best for: semantic similarity, cold-start items, cross-domain transfer. Unique advantage: understands meaning, not just keyword overlap.

Automated Model Selection Pipeline

Zero manual intervention from data upload to model deployment. The pipeline handles everything — from feedback detection to model promotion.

01  Feedback Detection
FeedbackDetector inspects dataset columns to classify as explicit (rating column present) or implicit (binary events only). Filters the algorithm pool so that SVD/NMF only run on explicit datasets, while ALS/BPR only run on implicit. Prevents nonsensical model evaluations.
02  Scale Tier Classification
DatasetAnalyzer counts unique users and items. Tier-A (>10K users or >5K items): downsamples to Tier-A limits before memory-intensive algorithms. Tier-B (>3K/2K): applies lighter downsampling. Prevents OOM errors on algorithms with quadratic memory complexity.
03  Algorithm Eligibility Filter
Algorithms with hard scale limits (max_users, max_items) are checked against post-downsampled dataset dimensions. Ineligible algorithms are skipped with a recorded skip_reason in the training result, maintaining a complete audit trail of which models were excluded and why.
04  Optuna Hyperparameter Tuning
For each eligible algorithm, Optuna TPE (Tree-structured Parzen Estimator) sampler runs N trials auto-budgeted by algorithm complexity. Best hyperparameters per algorithm are stored in a SQLite-persisted study (results/optuna_studies.db). TimeoutCallback limits total tuning to OPTUNA_TIMEOUT_S (default: 600s).
05  BenchmarkEngine Evaluation
BenchmarkEngine.run() evaluates each algorithm on a held-out test set. Computes NDCG@K, Precision@K, Recall@K, RMSE (explicit), MAE (explicit), training time, and scalability_score. Full leaderboard stored in last_training_result.json with per-algorithm detail.
06–07  Composite Scoring & Promotion
Composite = 0.7 × normalize(primary_metric) + 0.2 × normalize(speed) + 0.1 × scalability_score. Top-ranked model is automatically promoted to tenants/{id}/models/ and its model_id flagged as 'promoted' in registry.json. The next /recommend request instantly uses the new model — zero manual intervention required.

AI / ML Engineering

Feature Engineering & Embeddings

Transforming raw interaction logs and item metadata into rich representations that fuel every recommendation algorithm in the registry.

User Features
Interaction history, click sequences, explicit ratings, implicit signals (views, time-on-item), session context, and frequency patterns.
Item Features
Category hierarchy, price point, temporal attributes (release date, freshness), tags, genre, and domain-specific attributes.
Metadata Features
Text descriptions (TF-IDF weighted), structured attributes, entity relationships, and cross-domain taxonomy mappings.
Behavioral Features
Recency-weighted interaction signals, temporal decay functions, purchase vs. browse signals, and negative implicit feedback (skips, dismissals).

Interaction Matrix Construction
All interaction data is normalized into a scipy.sparse.csr_matrix — Compressed Sparse Row format. This reduces memory from O(U×I) dense to O(nnz) sparse, critical for datasets with <1% interaction density. The sparse matrix is the universal input format for all 21 algorithms.

Security Architecture

Security Controls & Gaps

Dual-auth, scope enforcement, and per-tenant isolation form a solid security baseline — with a clear path to production hardening.

Implemented Controls
JWT Authentication
HMAC-SHA256 signed tokens via python-jose. Configurable TTL. In-memory revocation list for token invalidation.
API Key Authentication
Per-tenant keys with hash-based comparison. TenantManager.resolve_api_key() maps hashed key to tenant_id — raw keys never stored.
Scope-Based Authorization
11 scopes from recommend:read to admin:write. @require_auth(scopes=[...]) decorator enforces per-endpoint authorization.
Tenant Isolation
All model loads use tenant_id prefix: tenants/{id}/models/. Cross-tenant model access is impossible at the API level.
BCrypt Password Hashing
passlib[bcrypt] with appropriate work factor. No plain-text credentials anywhere in the system.
Redis Rate Limiting
INCR counter with TTL per API key. Metering middleware tracks per-tenant call counts for billing and abuse detection.
Known Gaps — Production Roadmap
Secret Key in .env
AUTH_SECRET_KEY and REDIS_PASSWORD in repo .env. Migrate to AWS Secrets Manager, HashiCorp Vault, or GitHub Secrets for production.
In-Memory Token Revocation
_TOKEN_REVOCATION_LIST lost on restart. Move to Redis with TTL for distributed, persistent revocation across replicas.
No HTTPS Enforcement
TLS termination not configured in Docker Compose. Add NGINX reverse proxy with Let's Encrypt for production.
LLM-Generated SQL Risk
Smart DB CSV Builder constructs SQL from LLM output. Validate with sqlparse or enforce parameterized queries before execution.
No API Key Rotation
No built-in rotation mechanism. Add POST /tenant/rotate-key endpoint with grace period for smooth key rotation.
No MFA
Login uses only username + password. Add TOTP-based MFA for admin:write scope users in regulated industries.

Storage & Serving Bottlenecks
Filesystem-based tenants/ — no horizontal scaling without shared NFS/S3 Single FastAPI process — limited by Python GIL for CPU-bound scoring Model .pkl files grow unbounded — no automatic pruning policy Per-API-key rate limiting — not yet per-tenant quota management
Scaling Roadmap
Horizontal API Scaling
Multiple FastAPI replicas behind NGINX ALB. Redis for shared rate limiting and session state across replicas. Each replica independently loads models with optional Redis model cache.
Dedicated Model Serving
Replace .pkl in-memory with Triton Inference Server or BentoML. Separate recommendation service from training service for independent autoscaling. FAISS as a 5–10× faster ANN alternative to OpenSearch.
Storage Migration
S3/GCS for tenant model artifacts (multi-instance access). PostgreSQL for tenant metadata, job history, audit logs. Model artifact pruning policy: keep last N versions per algorithm per tenant.
